# Windows UTF-8 Guard

## 目标

在 Windows 环境中处理中文乱码或编码相关问题时，先判断根因，再做最小修复，避免把“显示问题”修成“文件损坏”。

## 适用场景

- 中文注释或中文文档显示成乱码
- PowerShell 里看起来乱码，但编辑器里正常
- 文件保存后突然出现大量乱码
- 仓库里混用了 UTF-8、UTF-8 with BOM、GBK、ANSI
- CRLF / LF 或 BOM 改动引发无关 diff

## 核心原则

1. 先区分“文件内容损坏”还是“终端显示错误”。
2. 先看单个目标文件，不要上来整仓库批量转码。
3. 保留原有换行风格和无关内容，避免修编码时制造额外噪音。
4. 在 Windows 上优先使用 PowerShell、编辑器状态栏或可查看原始字节的工具确认编码。
5. 只有在确认根因后，才修改 BOM、编码或换行风格。

## 排查流程

### 1. 确认现象

- 哪些文件乱码
- 在哪里乱码：编辑器、终端、Git diff、应用运行时，还是网页显示
- 最近一次出现乱码前做过什么操作

### 2. 判断问题类别

优先判断属于哪一类：

- 文件本身已经写坏
- 文件没坏，只是 PowerShell / 控制台按错误编码显示
- shell 重定向或脚本写入时破坏了编码
- 编辑器自动转码或自动换行导致 diff 膨胀

### 3. 检查文件字节与头部

必要时可用 PowerShell 检查字节头，例如：

```powershell
Get-Content -LiteralPath 'path\\to\\file.md' -Encoding Byte -TotalCount 16 | Format-Hex
```

常见判断点：

- `EF BB BF`：UTF-8 with BOM
- 无 BOM 不代表有问题，但在 Windows PowerShell 5.1 下更容易被误读

### 4. 交叉验证

至少做两种交叉验证：

- 用编辑器查看编码状态
- 用原始字节或十六进制头部查看
- 在不同读取方式下比对内容是否一致

### 5. 最小修复

仅在确认根因后选择动作：

- 如果是终端误读：优先修读取方式，不急着改文件内容
- 如果是无 BOM 的 UTF-8 中文文件在 Windows PowerShell 5.1 下频繁误读：可考虑改为 UTF-8 with BOM
- 如果文件确实被错误转码：先备份，再对单文件修复
- 如果问题来自 `>`、`>>`、`echo >` 等写法：后续避免继续使用

## 明确禁止

- 未确认根因前整仓库批量转 UTF-8
- 顺手统一所有文件 BOM
- 顺手统一所有文件换行风格
- 把终端显示问题误判成文件损坏

## 输出要求

完成处理时，明确说明：

1. 根因判断
2. 检查依据
3. 实际修改了哪些文件
4. 修改了编码、BOM 还是读取方式
5. 还有哪些风险或后续建议
