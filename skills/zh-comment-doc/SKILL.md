# 中文注释与文档

## 目标

让代码和文档更容易被中文团队维护，同时避免低质量注释污染代码。

## 基本规则

1. 注释和文档默认使用简体中文。
2. 注释写“为什么”和“这里容易错在哪里”，不写“这行代码表面上做了什么”。
3. 文档优先解决真实使用问题，不堆背景废话。
4. 技术名词、框架 API、命令行参数保留原始写法，解释部分用中文。

## 什么时候该写注释

- 核心流程
- 业务约束
- 重要边界条件
- 容易误解的性能取舍
- 非直觉实现
- 兼容处理或历史包袱仍暂时保留的地方

## 什么时候不该写注释

- 显而易见的赋值、判断、循环
- 仅仅重复方法名或变量名
- 会快速过时、又没有维护价值的描述

## 什么时候该补文档

- 增加或调整配置项
- 变更启动方式、部署方式或迁移步骤
- 引入新的使用约束
- 用户后续大概率会反复问同一个问题

## 工作流程

1. 先判断哪些位置真正需要注释或文档。
2. 用最少的文字说明最关键的信息。
3. 检查注释是否已经过时、空泛或与代码不一致。
4. 如果现有英文文档只有少量关键段落需要同步，优先补中文说明，不要盲目整份双语化。

## 输出要求

完成后说明：

1. 新增了哪些中文注释
2. 新增或更新了哪些中文文档
3. 为什么这些位置值得写
4. 哪些地方刻意没写，为什么
