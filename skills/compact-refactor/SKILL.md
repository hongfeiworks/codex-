# Compact Refactor

## 目标

执行偏紧凑、偏实用的重构：让代码更清晰，而不是更花哨。

## 适用场景

- 现有实现被拆得太碎
- 一堆只调用一次的小方法影响阅读
- 变量、中间层、包装函数过多
- 需要清理流程，但不想大改架构

## 重构原则

1. 主流程优先。阅读时应尽量一眼看清数据如何流动。
2. 不要为了形式化而抽方法。只调用一次、内容很短、没有复用价值的逻辑，优先内联。
3. 只有在以下情况成立时才抽方法：
   - 逻辑复用
   - 明显降低局部复杂度
   - 提升可测试性
   - 隔离副作用、错误处理或边界判断
4. 删除重复包装、无意义中间变量和陈旧兼容分支，但不要动无关模块。
5. 命名清楚比结构花哨更重要。

## 建议流程

1. 先找主流程和关键分支。
2. 标出这些“低收益抽象”：
   - 只调用一次的小方法
   - 纯转发包装
   - 没有信息增量的中间变量
   - 把本该顺着读的逻辑切碎的帮助函数
3. 判断哪些逻辑可以直接回收到主流程。
4. 修改后再检查：
   - 主流程是否更直
   - 方法数量是否下降
   - 单个函数是否变得失控臃肿

## 注释规则

- 如需注释，统一使用简体中文。
- 只注释关键流程、边界条件和不直觉实现。
- 不写把代码翻译一遍的注释。

## 输出要求

完成后说明：

1. 哪些逻辑被内联
2. 哪些方法被保留，为什么
3. 哪些方法被新建，为什么必须抽出
4. 做了哪些验证
