# 验证后再结束

## 目标

在宣布任务完成前，先证明修改真的生效，而不是靠主观判断收工。

## 适用场景

- 修 bug
- 重构
- 配置调整
- 文档或脚本改动影响使用流程
- 任何可能引发回归的代码修改

## 核心原则

1. 验证是交付的一部分，不是可有可无的附加项。
2. 验证方式和风险等级匹配：
   - 小改动做聚焦验证
   - 共享逻辑改动扩大验证范围
3. 如果无法验证，必须明确说明原因和剩余风险。

## 验证流程

### 1. 明确预期

- 这次修改后什么应该变好
- 什么行为不应该被改坏

### 2. 选择最合适的验证方式

按场景选择一种或多种：

- 运行相关测试
- 运行构建、类型检查、lint
- 复现原始问题并确认消失
- 执行关键命令或脚本
- 对用户可见流程做最小人工验证

### 3. 记录结果

至少说明：

- 跑了什么
- 结果如何
- 是否还存在没覆盖到的风险

### 4. 无法验证时的处理

如果环境、依赖或权限限制导致不能验证：

- 说明具体卡点
- 给出最小可执行的人工验证步骤
- 不要把“没条件验证”写成“已经没问题”

## 明确禁止

- 只改代码不验证就说完成
- 只看静态代码就断言运行正确
- 用无关测试通过替代关键路径验证

## 输出要求

结束前统一给出：

1. 预期行为
2. 实际验证动作
3. 验证结果
4. 未覆盖风险
5. 如需人工补测，给出最小步骤
