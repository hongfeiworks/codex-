# debug-root-cause-zh

请按“先定位根因，再修复”的方式处理下面的问题：

`$ARGUMENTS`

## 调试原则

1. 不要先拍脑袋改代码，先确认现象、影响范围和复现条件。
2. 区分症状、触发条件和真正根因，不要把表面报错当结论。
3. 优先收集证据：
   - 报错信息
   - 调用链
   - 输入输出差异
   - 最近相关改动
   - 环境差异
4. 先给出可检验的根因假设，再通过代码、日志、测试或最小复现验证。
5. 修复时优先最小化改动，避免顺手引入额外重构。
6. 修复后必须验证问题是否真的消失，并说明还有没有残余风险。

## 建议流程

1. 复述问题和当前已知事实
2. 列出最可能的 1 到 3 个根因假设
3. 逐项验证并收敛到真正根因
4. 实施最小修复
5. 用测试、构建、脚本或手动步骤验证

## 输出要求

按下面结构输出并执行：

1. 现象与影响范围
2. 根因假设
3. 验证过程
4. 修复方案
5. 验证结果
6. 剩余风险或后续建议
